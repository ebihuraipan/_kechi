# .gitプロジェクト内に複数のプロジェクトフォルダがある場合、gitのルートをvsocdeで開くんじゃなくて個別のプロジェクトを開くほうが良いか確認＆レポートを

## エグゼクティブサマリー

Copilotのトークン節約を優先するなら**個別プロジェクトで開くほうが有利**。ただしVSCodeの「Multi-root Workspace」（`.code-workspace`）を活用すれば、複数プロジェクト管理の利便性を保ちながらCopilotのコンテキストを最小化できる。

---

## 背景・問題

gitのルートに複数のサブプロジェクトが存在するモノレポ構成や、便宜上まとめたリポジトリでは、VSCodeをどの粒度で開くかによってCopilotの動作が変わる。

```
/repo-root/
├── project-a/
├── project-b/
└── project-c/
```

---

## gitルートで開く場合の影響

Copilotはワークスペースルート以下のファイルを参照対象とする。`/repo-root/`で開くと：

- `@workspace`の検索範囲がproject-a・b・c全体になる
- 現在の作業と無関係なファイルがコンテキストに混入する
- インデックスが大きくなり、Copilotの推論精度が下がる場合がある
- `.copilotignore`でプロジェクトごとの除外を書く必要があり管理が煩雑

---

## 個別プロジェクトで開く場合の効果

`/repo-root/project-a/`のみをVSCodeで開くと：

- Copilotのコンテキスト範囲がproject-aだけに絞られる
- `@workspace`の精度が上がり、関連ファイルが正確にヒットする
- `.copilotignore`の設定がシンプルになる（そのプロジェクト内の除外だけ書けばよい）
- 不要なタブが混入しにくい

---

## Multi-root Workspaceという選択肢

VSCodeの`.code-workspace`ファイルを使うと、複数プロジェクトを選択的に追加しつつ、各フォルダ配下のみをインデックスできる。

```jsonc
// mywork.code-workspace
{
  "folders": [
    { "path": "project-a" },
    { "path": "project-b" }
  ]
}
```

**メリット**: 複数プロジェクト間のファイル移動・Git操作が1ウィンドウで完結する。

**デメリット**: `@workspace`の範囲は追加したフォルダ全体になるため、gitルートで開くよりはましだが、個別プロジェクト単体より広い。Copilotトークン節約の観点では個別プロジェクト単体が最も有利。

---

## Copilotトークン節約の観点からの評価

| 開き方 | コンテキスト範囲 | トークン節約 | 利便性 |
|--------|----------------|------------|--------|
| gitルート全体 | 全プロジェクト | 低 | 高 |
| .code-workspace（一部） | 選択プロジェクトのみ | 中 | 中 |
| 個別プロジェクト単体 | 1プロジェクトのみ | 高 | 低（切り替え必要） |

---

## 結論・推奨

1. **基本は個別プロジェクト単体で開く**（Copilotトークン節約最優先）
2. 複数プロジェクトをまたぐ作業が多い場合は`.code-workspace`で必要なプロジェクトだけ追加する
3. gitルートをそのまま開くのは避ける（Copilotコンテキストが無制限に広がる）

---

## 補足：Copilot以外のメリット

個別プロジェクトで開くことはCopilot以外の面でも有益：

- VSCodeの検索（Ctrl+Shift+F）がノイズなく動作する
- 拡張機能の設定（`.vscode/settings.json`）がプロジェクトごとに独立する
- プロジェクト固有の`.editorconfig`・linter設定が正しく適用される
