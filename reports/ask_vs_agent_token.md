# GitHub Copilot の Ask Mode は Agent Mode よりトークン効率が良い

> ソース: [Copilot ask, edit, and agent modes: What they do and when to use them](https://github.blog/ai-and-ml/github-copilot/copilot-ask-edit-and-agent-modes-what-they-do-and-when-to-use-them/)  
> GitHub Docs: [Requests in GitHub Copilot](https://docs.github.com/en/copilot/concepts/copilot-billing/copilot-requests)

---

## Ask / Edit / Agent 各モードの概要と違い

### Ask Mode（質問モード）

コードや技術的な疑問を投げかけると、**ファイルには一切触れずに**回答だけを返す最もシンプルなモード。

- ハイライトしたコードの説明、テスト方法の提案、コードスニペットの提示などが得意
- プロジェクトへのコミットも、アーキテクチャへの影響もゼロ
- 「Tailwind のスタイリング方法は？」「JavaScript のクロージャとは？」など、調べものにも使える
- 自律的な動作は一切なく、1回の会話 = 1回のリクエスト消費

### Edit Mode（編集モード）

変更対象のファイルをユーザーが明示し、自然言語で指示を与えると、**差分（diff）を表示した上でユーザーの承認を待つ**モード。

- 「エラーハンドリングを追加して」「async/await でリファクタリングして」などの指示に従い直接コードを書き換える
- ただし **保存・適用前に必ずユーザーが diff を確認する**設計
- カスタムインストラクションで命名規則・スタイルを事前に指定できる
- 作業範囲がユーザー指定のファイルに限定されるため予測可能
- 1回の指示 = 1回のリクエスト消費

### Agent Mode（エージェントモード）

高レベルなプロンプト 1つを受け取り、**計画立案・ファイル選択・ツール呼び出し・ターミナルコマンド実行・反復修正を自律的に行う**最も強力なモード。

- プロジェクト全体を横断してコードを分析し、関連する変更を自動で複数ファイルに適用する
- Edit Mode とは異なり、**ユーザーの承認なしに編集を進める**（潜在的に危険なコマンドのみ確認あり）
- 「連続編集ドライバー」モデル：開発者がゴールを定義し、Copilot がステップを自律実行する
- 1回のユーザープロンプト = 1回のプレミアムリクエスト消費（ただしその後の自律ツール呼び出しは追加カウントなし）

## なぜ Agent Mode はトークンを多く消費するか

### 1. コンテキストウィンドウの大量消費

Agent Mode はプロジェクト全体を横断して推論する。関連ファイルの読み込み・参照・整合性チェックのために、**1 回の応答に乗せるコンテキスト量が Ask/Edit に比べて桁違いに大きい**。

### 2. 自律ループによるオーバーヘッド

Agent Mode の内部動作は「計画 → 実行 → 検証 → 修正」のループ構造になっている。各ループでシステムプロンプト・タスク説明・ツール定義・実行履歴がすべてコンテキストに積み上がっていく。これが繰り返されるほど、1 タスクあたりの総トークン数は膨らむ。

### 3. ツール定義の固定オーバーヘッド

Agent Mode ではファイル操作・ターミナル実行・コード検索などのツール定義が常にプロンプトに含まれる。Ask Mode ではこのオーバーヘッドが存在しない。

### 4. 多段階の自律判断

Agent Mode は指示を受けてから完了まで、Copilot 側が何度も「次に何をすべきか」を判断する。その判断ごとに入力トークン（これまでの全会話履歴 + ツール結果）を再度積んで推論するため、タスクが複雑なほどコストが累積する。

> **重要な補足（公式 Docs より）**: リクエストのカウント上は、Agent Mode でのツール呼び出し（自律アクション）はプレミアムリクエストとしてカウントされない。カウントされるのはユーザーが送ったプロンプトのみ。ただしトークン（コンテキスト）消費量そのものは増加するため、モデル乗数が適用される場合やクォータに接近している場合は影響が出る。

## Ask Mode を使うべき場面

| 場面 | 理由 |
|------|------|
| 「このコードが何をしているか説明して」 | 読み取り専用の質問はコンテキストが小さく済む |
| ライブラリの使い方・API の調べもの | 外部知識の引き出しはトークン効率が高い |
| コードの特定部分についてアドバイスが欲しい | ファイル全体をスキャンさせない |
| 作業中に詰まったとき「今すぐ答えが欲しい」 | 1回のやり取りで完結、最短経路 |
| センシティブなシステムの近くで作業しているとき | ファイルを一切書き換えないため安全 |
| 月末でプレミアムリクエストが残り少ないとき | 消費を最小限に抑えられる |

公式ブログの表現を借りると、Ask Mode は「エディタの中の静かなひとりごと」。答えをもらうだけで、プロジェクトへの関与はゼロ。

---

## Agent Mode を使うべき場面（使い分けの基準）

| 場面 | 理由 |
|------|------|
| 新規リポジトリの立ち上げ・スキャフォールディング | 複数ファイルを横断した一括生成が得意 |
| README を渡してアプリの骨格を作らせる | 高レベルプロンプト → 全体構築の典型用途 |
| 機能追加で複数ファイルの整合性維持が必要なとき | 関連変更を自動で追跡・適用できる |
| バグ修正の影響範囲が広いとき | プロジェクト横断の推論が強み |
| 繰り返し作業（ルーティングの整理、ボイラープレート削除など）| 自動ループで人手を減らせる |
| 明確・詳細な初期プロンプト + カスタムインストラクションがある場合 | 情報が揃っているほど Agent Mode の精度は上がる |

**使うべきでない場面**:
- センシティブな部分を「触らせたくない」ファイルがある場合
- 2〜3 ファイルの局所的な変更（Edit Mode の方が適切）
- 結果の予測可能性を最大化したいとき

## 節約の目安

### プレミアムリクエストのカウント（公式仕様）

Ask・Edit・Agent のいずれも、**ユーザーが送ったプロンプト 1回 = 1プレミアムリクエスト**（× モデル乗数）という点で同じ。Agent Mode の自律ツール呼び出しはカウントされない。

つまり**リクエスト数の節約という観点では、Ask と Agent に差はない**。

### ただしトークン消費の実態は異なる

リクエスト数が同じでも、Agent Mode は 1 リクエストあたりのトークン（コンテキスト）消費が大きい。これは以下に影響する：

- **応答速度**: コンテキストが大きいほど応答が遅くなる
- **コンテキストウィンドウ上限**: 長いセッションでは途中で情報が切り落とされるリスク
- **2026年6月以降の従量課金**: GitHub は 2026年6月1日よりリクエスト数ベースからトークン使用量ベースの課金に移行予定。この移行後は Agent Mode の実際のコスト差がより明確になる見込み

### モデル乗数（参考）

使用するモデルによってプレミアムリクエストの消費倍率が異なる。コスト節約には軽量モデルの選択も効果的。

| モデル | 乗数（有料プラン） |
|--------|----------------|
| GPT-4.1 / GPT-4o / GPT-5 mini | **0（無料）** |
| Claude Haiku 4.5 / Gemini 3 Flash / GPT-5.4 nano / GPT-5.4 mini | 0.25〜0.33 |
| Claude Sonnet 4.6 / Gemini 3.1 Pro / GPT-5.2 など | 1 |
| Claude Opus 4.5 / 4.6 | 3 |
| GPT-5.5 | 7.5 |
| Claude Opus 4.7 | 15 |

**ポイント**: 複雑なタスクを Agent Mode + 高コストモデルで実行すると、コンテキスト消費量×高乗数でコストが大きくなりやすい。シンプルな質問には Ask Mode + 無料モデル（GPT-4o等）を使うのが最も効率的。

## まとめ：使い分けの判断フロー

```
質問・調べもの？ → Ask Mode（最小コスト）
        ↓
変更先ファイルが明確で局所的？ → Edit Mode（コスト小、制御しやすい）
        ↓
複数ファイル横断・自律実行が必要？ → Agent Mode（コスト大、パワフル）
```

> **節約の基本戦略**: タスクの複雑さに合わせてモードを下げる。「Agent Mode でできるから」という理由だけで毎回 Agent を使わず、Ask → Edit → Agent の順で必要性を確認してから選ぶ。
